## 第二章 · 技术标准

### 2.1 URL 与跳转
- 一律 **www + 全小写 + clean URL**
- 裸域 → www **301**;www 返回 **200**
- 俄语 URL **无尾斜杠**(`/ru`、`/ru/products`)
- 🔴 **标签内 URL 必须直接 200**:canonical / hreflang / sitemap / 内链里写的每个 URL,直接访问不得触发任何 301/308(历史事故:尾斜杠问题返工两轮)

### 2.2 多语言(EN + RU)
- 新页面**EN 与 RU 同时产出**,由 Codex 直接产出,不需外部人工译文,不用停下等
- hreflang 三条 `en / ru / x-default`:**双向对称、每个 URL 直接 200**
- 🔴 **俄语页 canonical 指向俄语页自身**(指英文页 = 俄语页永不被索引,最危险的坑)
- `<html lang>` 正确;**俄语版内容必须与英文版对齐**,不得出现"英文有内容、俄语空白"
- 编码统一 **无 BOM UTF-8**(历史事故:旧 Windows 代码页把俄文与特殊标点永久写成 `?`)

### 2.3 结构化数据
- 产品页:**Product**(name / description / image=WebP / brand / sku / offers)。询价制统一 `price: "0.00"` + USD + InStock,**不得放具体价格**
- 分类页:**CollectionPage + ItemList**
- 文章/指南页:**Article**;凡有 FAQ 区块必须加 **FAQPage**(AEO 关键)
- 表格用真实 HTML `<table>`,**不得用图片代替**(搜索引擎与 AI 要读得到)

### 2.4 图片
- 上传前 **PNG → WebP**;og:image 用 WebP
- 各站图片走**各自自有 CDN**(images.<各自域名>),🔴 **不得跨站引用其他站 CDN**
- 主图 `object-fit: contain` **完整显示不裁切**(历史事故:海报硬塞网格导致图乱)
- 懒加载 + `content-visibility: auto`;alt 必须真实描述
- **pHash + dHash 跨站扫描**:新图预期 0 碰撞,有命中 → **停下报告**,不自行处理
- **OCR 图内文字防线**:上传预检 + 每周扫。违禁词(价格 / 认证 / 旧数据 / 区间MOQ)必撤;`Low MOQ` 等模糊表述属可豁免,豁免须绑 **SHA + pHash 双哈希**(图变即失效)

### 2.5 页面结构标准(产品页 = CBF 结构)
干净主图(contain 不裁切)→ 真实 HTML 规格表 → Key Features → Colors → 海报降级灯箱画廊 → 询价口径块 → 询盘 CTA。
🔴 **不得铺海报网格**(会导致裁切错位)。文字层必须是真实 HTML(吃 SEO/AEO),不得以图代文。

### 2.5b 🔴 卡片缩略图(列表卡 / 分类卡 / 展示位卡 / 博客卡)
- 🔴 **不得直接用整张营销海报缩小当缩略图** —— 海报是为详情页大图设计的,信息密度高,缩小后文字与细节全糊,买家看不清产品(事故:Junyi 产品列表卡与分类卡全部糊成一片)
- **应使用干净的产品主体图**:若无独立白底产品图,则从海报中**裁出产品主体区域**做衍生缩略图
- **裁切要求**:只保留产品主体,尽量不含标题/规格表/色卡/图标等密集信息;🔴 **不得裁掉产品本身**;背景干净;输出 WebP 且尺寸适配实际显示
- 🔴 **详情页画廊仍用完整海报原图**;衍生缩略图不替代、不减少产品原始图集
- 🔴 **不得使用非专业图片**:翻拍电脑屏幕、含桌面杂物/反光的随手拍等一律不可用于对外页面(事故:Junyi 博客卡曾用翻拍显示器照片,且两张重复)
- 同一区域内多张卡片**图片不得重复**

### 2.6 悬浮联系球(所有站统一)
- 右下角常驻;**WhatsApp 官方 SVG 图标**(禁 `WA` 等文字缩写);桌面 hover 显示 `Chat on WhatsApp`;`aria-label="Contact us on WhatsApp"`
- 轻点=打开 WhatsApp;长按/拖动=移动(**10px 位移阈值**防误触);松手**吸附最近左右边缘**;限可视区内;移动端同样常驻小球
- **水波纹:3 圈 / 单圈 3s / delay 0-1-2s 错开 / 最大 3.2 倍 / 描边 5px / #25D366**;禁细线、淡光圈、实心呼吸盘、快速闪烁
- 🔴 **波纹必须放独立伪元素或子层**(`::before` / `::after` / 独立 span),**绝不可在可拖拽按钮本体上做 transform 动画**(历史事故:transform 冲突导致动画被覆盖,连续两次"报告通过但线上静止")
- `prefers-reduced-motion` 下三层波纹 `display:none`

## 第三章 · 工作方式

### 3.1 交付要求
每个任务交付时必须包含:
1. 做了**哪些判断及理由**(不只是"做完了")
2. **任何停下报告的项**及原因
3. 涉及页面的**最终 URL**
4. `seo:audit` 状态

### 3.2 遇冲突先停
- 指令与本宪法冲突、或指令本身有歧义、或发现意料外的问题 → **停下报告,不自行放行**
- 🔴 **需要凭据时必须停**:API key、邮箱密码、账号密码、验证链接 —— **一律由运营者本人操作**,Codex 不代填、不代注册

### 3.3 动效 / 视觉类验收(不接受口头"通过")
静态截图无法判断的动效、交互,**必须提供可独立验证的证据**:
- Production 正式网址(不用特殊 query)+ 无痕模式验证
- 连续截 20–30 帧(间隔 100ms)合成 GIF,肉眼可确认
- 或 F12 `getAnimations()` 输出证明动画 running + duration + delay
(历史事故:悬浮球水波纹两次"双帧采样确认在动",线上实际静止,自检采样采错对象)

### 3.3b 🔴 防"假成功"(自动化流程必须验到底)

任何自动化流程(部署、缓存清理、推送、扫描)**不得以"调用没报错"当作成功**。判定绿灯必须**同时满足全部实际成功条件**,任一不满足即失败。

**范例(Cloudflare 缓存清理,2026-07-23 落地)**:必须同时满足
`HTTP 2xx` + `ok=true` + `cloudflareSuccess=true` + **非 skipped**,否则直接判失败。

**通用要求**:
- 不能只看 HTTP 状态码 —— 接口返回 200 但业务失败(如鉴权错误、被跳过)是常见陷阱
- 不能把"跳过/未执行"当成功
- 凭据配置类失败必须**暴露真实原因**(如 Token 错误导致 401),不得静默吞掉
- 🔴 **凭据泄露处理**:任何 Token/Key 若曾出现在截图、日志或对话中 → **立即作废并重建**,新凭据遵循**最小权限**(只授权必需的区域/范围)

> **事故来源**:CWB Cloudflare 清缓存长期"看似配置好了"实为误填其他平台密钥(`sk_live_` 开头)导致 401;修复时同步删除了截图泄露的旧 Token,并把 Token 权限收窄到单一区域。

### 3.4 改动已上线站要逐站回归
任何触碰已验收上线站(nameer / junyi / CBF)的改动,必须**逐站回归测试**,确认既有功能(联系方式、邮箱明文、语言切换、audit)未被破坏。

### 3.5 判断尺度
- **模糊 ≠ 错误**:`Low MOQ` 可豁免;`MOQ 100 PCS` 必撤
- **一次性例外 ≠ 常规承诺**:如"疫情期间 15 天空运"只能写成"那一次",不得写成常规能力
- **发现问题 ≠ 制造问题**:清理历史欠账是对的,主动报告自身疏漏是加分项
- **不为交差编证据**:识别误报不误删,诚实说明工具局限,比"全绿报告"更有价值

## 第四章 · 增量原则与展示位管理

### 4.1 🔴 增量原则(改哪儿动哪儿,别的不碰)

**核心**:每次任务**只动本次范围内的东西**,已上线且验收通过的内容一律不重做。

- 🔴 **已上线的产品页/内容页,不在本次任务范围内的一律不动** —— 不重新生成、不重写文案、不重传图片
- 🔴 **框架/样式/组件调整只改模板层**,产品数据(规格、文案、图片、URL、schema)**不重新生成**
- 🔴 **已发布的 URL 不得变更**(改了等于丢掉已积累的收录与外链)。确需变更 → 停下报告,由运营者拍板并规划 301
- 若某次改动**确实必须触碰**已上线内容 → **停下报告**,说明:为什么必须动、影响哪些页、风险是什么,**等运营者确认后再动**
- **理由**:产品与内容会越积越多,每次调整若重跑全部,既浪费时间又可能改坏已验收的东西

### 4.2 发现问题时:报告 + 横向排查(🔴 默认动作,不需要每次被要求)

自查或执行中发现范围外的问题(不是本次任务造成的):
1. **报告**给运营者:问题是什么、在哪、影响多大
2. 🔴 **默认横向排查所有站点**:同一问题在其他页面/其他站是否也存在(例:发现某页邮箱被 Cloudflare 混淆 → 横扫全站与其他站,找出全部)。**不论问题在哪个站被发现,都要查全部站,不只修发现的那一处**
3. **一并列出清单**,说明建议怎么修
4. **修不修、什么时候修 → 运营者决定**;紧急红线问题(如邮箱混淆、价格泄漏、认证词出现)可直接修并报告

> **正面先例**:2026-07-21 Codex 发布博客时,主动发现 nameer 8 个产品页的 Cloudflare 邮箱混淆复发,横扫定位并修复 —— 这是期望的行为。

### 4.2b 🔴 规则沉淀(每次踩坑必做)

任何新发现的坑、新定下的标准,**必须当场沉淀,不能只停留在一次修复**:
1. **修复**当次问题
2. **写入本宪法**(或 SOP)成为永久规则,注明触发它的真实事故
3. **给 lint / audit 加自动检查规则**(能自动化的一律自动化)
4. **同步所有站**的宪法文件,确保版本一致

> **意义**:规则进流水线,人就不用记。同一个坑不踩第二次。

### 4.3 首页/展示位轮换规则(防止内容越堆越多)

**问题**:新品不断上架,若展示位只增不减,首页会越堆越长、图片越存越多,导致站点臃肿、后续调整成本升高。

**规则**:
- **展示位数量固定**(各站在建站/本规则落地时确定并写入代码注释;**Codex 需报告各站当前实际展示位数量**,如首页精选 N 款、分类页封面 N 张)
- **按上架时间倒序**:新品自动进入展示位,**最旧的自动被顶出**,不再展示
- **被顶出 ≠ 下架**:产品页本身**继续存在、继续被收录**,只是不再占用首页展示位
- 🔴 **选图必须做,不能跳过**:每批产品上架时,**必须同时确定首页/展示位用图**。若助手已在指令中指定用哪张 → 照办;**若指令未指定 → Codex 必须自行选定,这是最后一道关,不得留空**

**选图标准(按欧美 B2B 买家审美)**:
- 首页/展示位优先选**生活方式实拍或干净产品主体图**,画面清爽、主体突出、无密集文字
- 避免选信息密集的规格拼图、参数表类海报作为展示位图
- 需要时可**裁切**出干净主体,但🔴 **裁切不得裁掉关键卖点文字或产品主体**
- 各站展示位视觉风格保持统一,不因单款海报配色不同而显得杂乱

**🔴 展示位与产品页的关系(务必理解正确,别弄反):**
- 展示位显示的图,是**引用**(reference)CDN 上的同一份文件,**不是把图"移动"或"搬"到首页**
- 🔴 **产品页的完整图集永远不动**:每款上传了几张,产品页就一直有几张。**无论该款是否进入展示位、是否被顶出,产品页图片一张都不能少**
- 同一张图**只存一份**在 CDN,首页与产品页各自引用同一地址;展示 ≠ 占用 ≠ 转移
- 被顶出展示位,只是**不再在首页显示**,产品页照常完整展示、照常被收录

**图片资产清理(防止越存越多)**:
- **被顶出展示位时**:
  - 若该图是**为展示位专门裁切/重做的衍生图**(不属于产品原始图集)→ **删除**(含 CDN/R2 存储),并从 dedup 基线与 OCR 白名单移除
  - 若该图是**直接引用产品页原图** → 🔴 **绝对不删**(产品页仍在使用,删了产品页就缺图)
- 🔴 **任何清理动作前必须确认:该图是否仍被任何页面引用。有引用一律不删。**
- 定期(建议随每周 audit)检查:是否存在**无任何页面引用的孤儿图片**,列出并报告,经运营者确认后清理
